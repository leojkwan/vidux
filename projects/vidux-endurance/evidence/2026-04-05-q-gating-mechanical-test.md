## Surprise Finding: `[Depends: none]` False Positive

During initial testing, both tasks included `[Depends: none]` annotations. This caused a false blocker:

```json
{
  "task": "Task A: Implement feature X [Evidence: test] [Depends: none]",
  "blocked": true,
  "action": "blocked",
  "context": "Waiting on: none"
}
```

**Root cause (line 124 of vidux-loop.sh):**
```bash
if grep -E '^\- (\[ \]|\[(pending|in_progress)\]) ' "$PLAN" | grep -qi "$DEP_TARGET"; then
```

When `[Depends: none]` is parsed, `DEP_TARGET="none"`. The case-insensitive grep then matches the literal string "none" inside OTHER pending task lines that also contain `[Depends: none]`. This is a self-referential match — the dependency checker finds "none" in peer tasks' annotations and incorrectly flags the task as blocked.

**Impact:** Any plan using `[Depends: none]` as a "no dependency" marker will false-positive block tasks whenever there are 2+ pending tasks with that annotation. The workaround is to omit the `[Depends:]` annotation entirely when there is no dependency. A code fix could add a special-case for `none` (e.g., `if [ "$DEP_TARGET" = "none" ]; then skip;`).

**Severity:** Low-medium. Plans generated by LLMs commonly use `[Depends: none]` as a placeholder, so this could surface in real usage.
